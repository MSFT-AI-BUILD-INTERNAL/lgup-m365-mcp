APIM은 Consumption(서버리스) 티어로 배포되며, mcp라는 단일 API 아래 4개의 작업을 노출합니다. 보안의 핵심은 POST /mcp 작업에 걸린 validate-jwt 정책입니다.

5.2 POST /mcp 의 validate-jwt 정책 (현행)

APIM의 <required-claims><claim name="scp">는 공백으로 구분된 scp 문자열을 분할하지 않고 통째로 비교합니다. 따라서 토큰의 scp가 "access_as_user user_impersonation"처럼 다중값이면 매칭에 실패합니다. 이를 피하기 위해 JWT 검증은 validate-jwt로, 스코프 검사는 choose에서 scp를 분할하여 수행합니다. 

<validate-jwt header-name="Authorization" require-scheme="Bearer"
    failed-validation-httpcode="401"
    failed-validation-error-message="Unauthorized. Valid Entra bearer token required."
    output-token-variable-name="jwt">
  <openid-config url="https://login.microsoftonline.com/<tenantId>/v2.0/.well-known/openid-configuration" />
  <audiences>
    <audience>api://<clientId></audience>
    <audience><clientId></audience>
  </audiences>
</validate-jwt>
<choose>
  <when condition="@(!((Jwt)context.Variables[&quot;jwt&quot;])
      .Claims.GetValueOrDefault(&quot;scp&quot;,&quot;&quot;)
      .Split(' ').Contains(&quot;access_as_user&quot;))">
    <return-response>
      <set-status code="403" reason="Forbidden" />
      <set-body>{"error":"Forbidden. Token must include the 'access_as_user' scope."}</set-body>
    </return-response>
  </when>
</choose>
현황 반영 (2026-06-30) 라이브 APIM과 modules/gateway.bicep 모두 위 정책으로 정렬되었습니다. 또한 Entra 앱 92ad2fc4…에 access_as_user 위임 스코프를 추가(기존 user_impersonation 유지)하고, Azure CLI 클라이언트를 두 스코프에 사전 승인했습니다. subscriptionRequired=false(구독키 불필요)와 디스커버리 작업 2개도 적용되어 있습니다.

6. Container Apps Easy Auth 계층

Container App에는 내장 인증(Easy Auth)이 authConfigs/current로 구성되어 있습니다. APIM을 통과한 토큰을 한 번 더 독립적으로 검증하고, 검증된 신원을 백엔드 컨테이너에 헤더로 주입합니다.

구성 요약

  • platform.enabled = true
  • unauthenticatedClientAction = AllowAnonymous
  • openIdIssuer = …/<tenantId>/v2.0
  • allowedAudiences = [ api://clientId , clientId ]

주입되는 신원 헤더

  • x-ms-client-principal-name (UPN)
  • x-ms-client-principal-id (oid)
  • x-ms-client-principal-idp (aad)
  • x-ms-client-principal (Base64 클레임 전체)
AllowAnonymous 의 의미 익명 호출도 백엔드까지 도달합니다(Easy Auth가 차단하지 않음). 유효한 토큰이 있을 때만 신원 헤더를 주입하며, 실제 접근 차단은 APIM(1차)과 앱의 scope_failure_response()(3차)가 담당합니다. 이 구조 덕분에 헬스 프로브와 디스커버리 엔드포인트가 토큰 없이도 동작합니다.

7. 애플리케이션 계층 방어 (scope_guard)

MCP 서버는 APIM·Easy Auth를 신뢰하되, 방어적 심층 검증으로 /mcp 진입 시 토큰을 한 번 더 확인합니다. main.py의 미들웨어 enforce_mcp_scope가 POST /mcp에서 identity/scope_guard.py의 scope_failure_response()를 호출합니다. 

# app/src/identity/scope_guard.py
# main.py 의 미들웨어 enforce_mcp_scope 가 POST /mcp 에서 호출
def scope_failure_response(authorization):
    token = AccessToken.from_authorization_header(authorization)
    # 1) Bearer 토큰이 없으면 401 + WWW-Authenticate (디스커버리 유도)
    if token is None:
        return JSONResponse(status_code=401, headers={
            "WWW-Authenticate":
                'Bearer realm="{application_id_uri}", '
                'authorization_uri="{authorization_endpoint}"'}, ...)
    # 2) scp/roles 에 access_as_user 포함 여부 확인 → 없으면 403
    if not token.has_scope("access_as_user"):
        return JSONResponse(status_code=403, ...)
    return None  # 통과 (다음 핸들러로)
  • 401: Bearer 토큰 자체가 없음 → WWW-Authenticate 헤더로 OAuth 디스커버리 경로 안내.
  • 403: 토큰은 있으나 access_as_user 스코프가 없음.
  • 신원 해석: get_current_user 도구가 Easy Auth 헤더(우선) 또는 Bearer 클레임에서 사용자 정보를 조립.
✓ 스코프 정합성 (해결됨 2026-06-30) Entra 앱 92ad2fc4…에 access_as_user 위임 스코프를 추가하여 토큰이 해당 스코프를 포함하도록 정렬했습니다 (기존 user_impersonation도 유지). APIM은 choose에서 scp를 분할 검사하고, 앱의 scope_failure_response()도 동일하게 분할합니다. 다중 스코프 토큰("access_as_user user_impersonation")에서도 정상 통과합니다.

그림 4 — MCP 서버가 외부 MCP 서버를 호출할 때의 신원 전파 패턴

8.1 신원 전파 전략 비교

전략방식적합한 경우
A. On-Behalf-Of (OBO) 수신한 사용자 토큰을 Entra의 OBO 흐름으로 다운스트림 api://…/.default 토큰으로 교환 다운스트림이 사용자별 권한을 요구할 때(권장)
B. Managed Identity Container App의 사용자 할당 관리 ID(lgmcp-dev-uami)로 앱-앱 토큰 획득 다운스트림이 서비스 신원만 필요할 때
C. API Key / Key Vault Key Vault에 보관한 키를 관리 ID로 읽어 헤더에 첨부 OAuth 미지원 레거시(NGIS/DRM 등)
보안 원칙 외부 MCP 호출용 비밀은 코드/파라미터 파일에 두지 말고 Container App 시크릿 또는 Key Vault로만 주입하세요 (auth-client-secret, ngis-api-key, drm-api-key 시크릿 참고). 관리 ID로 Key Vault에 접근하면 키 노출을 최소화할 수 있습니다.

8.2 DRM 복호화 — 직접 호출 연계

4.2 직접 호출 플로우의 백엔드 동작입니다. 업무 클라이언트가 Entra 토큰으로 APIM /mcp에 drm_decrypt 같은 도구를 호출하면, MCP 서버가 외부 DRM API(DRM_API_BASE_URL)를 호출해 복호화합니다.

구간인증/자격설명
클라이언트 → APIMEntra 토큰(access_as_user)APIM validate-jwt + scp 분할 검사 통과 필요
APIM → Easy Auth → MCP동일 토큰 + x-ms-client-principal*호출자 신원이 MCP 서버로 전파
MCP → DRM APIdrm-api-key(Key Vault) 또는 OBO복호화 요청. 키는 Container App 시크릿/Key Vault에서 주입
DRM 복호화 주의 복호화 결과는 민감 데이터입니다. ① 전 구간 TLS, ② 호출자 신원(x-ms-client-principal-id) 기반 인가·감사 로깅, ③ 복호화 데이터의 로그/캐시 금지, ④ DRM 키는 Key Vault + 관리 ID로만 접근을 권장합니다.

9. OAuth 디스커버리 & DCR

OAuth를 인지하는 MCP 클라이언트(예: Copilot Studio 동적 등록)는 인증 요건을 자동 발견할 수 있도록 두 개의 비인증 메타데이터 엔드포인트가 제공됩니다.

엔드포인트표준핵심 응답
/.well-known/oauth-protected-resource RFC 9728 resource: api://clientId, scopes_supported: [access_as_user], authorization_servers
/.well-known/oauth-authorization-server RFC 8414 Entra의 authorization_endpoint·token_endpoint·jwks_uri, code_challenge_methods: [S256]

401 응답의 WWW-Authenticate 헤더가 클라이언트를 이 디스커버리 경로와 authorization_uri로 유도하여, PKCE 기반 Authorization Code 흐름을 자동으로 시작하게 합니다.

10. 알려진 함정과 진단

함정 1 — 토큰 버전 불일치(403) 앱 등록이 v1 토큰을 발급(requestedAccessTokenVersion=null)하면 Easy Auth issuer(v2.0)와 불일치로 403.
해결: Microsoft Graph로 api.requestedAccessTokenVersion = 2 패치 → 토큰이 iss=…/v2.0, aud=<clientId>로 발급됨.
함정 2 — MSAL 토큰 캐시 Azure CLI/MSAL이 옛 v1 토큰을 캐시합니다. 앱 등록 패치 후에도 403이 지속되면 새 토큰을 강제 발급하세요.
rm로 ~/.azure/msal_token_cache.json의 AccessToken 항목 정리 후 재시도.
정상 동작 확인 방법 
# 1) 사용자 토큰 발급
az account get-access-token --scope "api://<clientId>/.default" --query accessToken -o tsv

# 2) 라이브 /mcp 호출 (get_current_user)
curl -s -X POST "https://<fqdn>/mcp" \
  -H "Authorization: Bearer $TOK" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json, text/event-stream" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call",
       "params":{"name":"get_current_user","arguments":{}}}'
# → authenticated:true, identityProvider:"aad", upn 표시되면 인증 성공
진단 팁
  • /.auth/me는 토큰 저장소 비활성화 시 인증이 정상이어도 403 → 무시하고 /mcp+get_current_user로 검증.
  • 토큰 없음 / 잘못된 aud(Graph·ARM 토큰) / 가짜 JWT → AllowAnonymous로 200(익명). aud가 맞는 토큰만 실제 검증.
  • allowedApplications:[]은 ARM이 주입하는 기본값으로 “테넌트 전체 허용”을 의미(제한 아님).
