Azure 구현 로드맵 — DRM 복호화 · 슬기 연동

최신 반영 (2026-07) — “현재 자산 인벤토리”가 갱신되었습니다(azure-implementation-roadmap.md). MCP 서버 앱은 골격이 아니라 DDD 구조로 구성되어 있으며, 테스트 UI(/auth-ui·/drm-ui), DRM 프록시(/drm/decrypt), OAuth 디스커버리(/.well-known/oauth-*)가 이미 구현되어 있고, Python 포팅(app-py/)도 존재합니다.

아키텍처 다이어그램의 Azure 영역(MCP/API 진입점 · DRM 호출 · 인증/키 관리 · 상암 데이터센터망/슬기 API 연동)을 실제로 구현하기 위한 단계별 로드맵입니다. 현재 자산(Bicep 인프라 + 최소 MCP 앱)과 목표 아키텍처의 격차를 정리하고, Phase 0~3의 실행 항목과 결정 필요 사항을 정의합니다.

0. 요약

인프라 뼈대(Container Apps · ACR · Key Vault · Storage · APIM · 관측성 · Easy Auth)는 약 70% 완성되어 있으나, 다이어그램이 요구하는 실제 업무 기능 (DRM 복호화 파이프라인, 클라이언트 파일 전달, 상암 DC 슬기 API 연동, 온프레미스 네트워킹)은 거의 미구현(0%) 상태입니다.

3. Gap 분석 (다이어그램 요소별)

A. MCP/API 진입점

  • 복호화 요청을 받는 툴/엔드포인트(decrypt_file 등) 부재.
  • 노출 방식(MCP/REST/둘 다) 미결 — 앱·APIM·커넥터 설계의 분기점.
  • 요청 단위 멱등성·작업 ID(비동기 시) 설계 없음.

B. DRM 호출 (복호화 핵심)

  • DRM API 스펙(인증·요청/응답 포맷·동기/비동기·파일 전달 방식) 미정의.
  • 복호화 파이프라인 단계(수신 → DRM 호출 → 결과 생성 → 반환) 미구현.
  • 대용량/타임아웃/재시도/실패 처리 정책 없음.

C. 인증/키 관리

  • 현재 자격증명은 env 시크릿 주입 의존 — Key Vault 런타임 조회 코드 없음.
  • Copilot Studio → Azure의 사용자 토큰 전달/On-Behalf-Of(OBO) 흐름 미설계.
  • 운영자/워크로드 RBAC 분리, 키 로테이션 정책 없음.

D. 상암 데이터센터망 / 슬기 API 최대 리스크

  • 온프레미스 연결 수단 없음 → VNet + ExpressRoute/S2S VPN + Private DNS 필요.
  • 현 APIM Consumption / 공개 Container Apps 구성은 사내망 도달 불가.
  • 슬기 API/MCP 클라이언트·엔드포인트·인증, 송신 IP 고정(NAT) 미정의.
